## 1. Purpose & Rebuild Scope

This document is the authoritative specification for rebuilding RiskMind on Replit. It defines architecture, design language, epics, per-screen UI specs, and all functional requirements with explicit MVP priority labels.

> **Replit Build Contract:** All requirements marked Must-Have are in scope for the initial Replit deployment. Should-Have items are built if time permits. Deferred items require a separate milestone decision.

### 1.1 What Is RiskMind

RiskMind is an AI-native, multi-organization enterprise risk management platform. It provides a single control plane for operational, compliance, and third-party risk with AI-assisted triage, Monte Carlo simulation, and automated monitoring.

## 2. Design Language & UI Specification

RiskMind uses a corporate-enterprise monochrome foundation. Color is a functional signal, not decoration. The single accent — Crimson — is reserved exclusively for critical risk states. All other communication is achieved through weight, scale, and greyscale contrast.

### 2.1 Design Philosophy

- Monochrome base — black, white, and five grey steps handle all structural UI.
- Crimson accent (#DC2626) is used only for: critical severity, KRI breach, overdue items, destructive actions.
- No gradients, shadows, or decorative color in any component.
- Typography hierarchy replaces color hierarchy wherever possible.
- Data density is preferred over whitespace — this is a professional operations tool.
- Progressive disclosure not oversaturated dashboards
- Apple-vibes not MS word 1997!

### 2.2 Color Tokens

| Token | Value | Usage |
|---|---|---|
| `--color-ink` | #000000 | Headings, borders, active nav, black badges |
| `--color-surface` | #FFFFFF | Page background, card background |
| `--color-gray-90` | #1A1A1A | Primary body text |
| `--color-gray-70` | #4D4D4D | Secondary body text, labels |
| `--color-gray-50` | #808080 | Placeholder, muted metadata |
| `--color-gray-30` | #B3B3B3 | Disabled states, divider lines |
| `--color-gray-10` | #E6E6E6 | Table borders, input borders |

| `--color-critical` | #DC2626 | Critical severity, breach, overdue |
| `--color-critical-bg` | #FEF2F2 | Critical badge background |

- Axes labeled: X = Likelihood, Y = Impact, both with numeric and text labels.

## 7. Data Model Requirements

### 7.1 Core Rules
- Every tenant-scoped table must include a `tenant_id` UUID column with NOT NULL constraint.
- Every table must include `created_at` and `updated_at` TIMESTAMPTZ columns with defaults.
- All primary keys are UUID (`gen_random_uuid()`).
- pgvector columns (`embedding vector(1536)`) on: risks, vendors, framework_requirements, signals.
- Drizzle ORM — all schema changes via migration files, never direct DDL.

### 7.2 Primary Tables

| Table | Key Columns | Notes |
|---|---|---|
| tenants | id, name, slug, settings | Root isolation entity |
| users | id, tenant_id, email, role, hashed_password | RBAC role stored here |
| audit_events | id, tenant_id, user_id, action, entity_type, entity_id, payload, created_at | Immutable; no updates or deletes |
| risks | id, tenant_id, title, category, owner_id, status, likelihood, impact, embedding | pgvector embedding column |
| treatments | id, tenant_id, risk_id, strategy, status, owner_id, due_date | Child of risks |
| kris | id, tenant_id, risk_id, name, warning_threshold, critical_threshold, current_value | Breach triggers alert |
| signals | id, tenant_id, source, content, status, classification, confidence, embedding | Pipeline state machine |
| vendors | id, tenant_id, name, tier, status, risk_score, contact_email | Lifecycle state machine |
| frameworks | id, tenant_id, name, version, type | Seeded: ISO 27001, SOC 2, NIST CSF |
| controls | id, tenant_id, title, description, status, owner_id | Many-to-many with requirements |
| alerts | id, tenant_id, type, severity, status, acknowledged_by, acknowledged_at, context | severity: critical\|high\|medium\|low |

## 8. Non-Functional Requirements

| Category | Requirement |
|---|---|
| Performance | p95 REST latency under 500ms for non-analytical reads. Analytical queries (simulation, gap analysis) are async. |
| Scalability | AI, simulation, and document processing tasks use BullMQ queues. Core CRUD remains synchronous. |
| Reliability | BullMQ jobs: exponential backoff retry (3 attempts), dead-letter queue for failed jobs. |
| Security | Strict RBAC + tenant RLS + bcrypt password hashing + signed magic link tokens (SHA-256, expiry-enforced). |
| Auditability | Immutable audit_events records for all state mutations. No audit record may be updated or deleted. |
| Extensibility | New frameworks seeded from JSON config. New MCP tools registered via NestJS module. New checks added to scheduler without core changes. |
| Testability | Service-level unit tests and integration tests per domain module. Tenant isolation tested with cross-tenant request scenarios. |
| Graceful Degr. | SMTP absent: notifications silently skipped. LiteLLM absent: AI features show 'manual mode' UI. Storage absent: file uploads return 503 with message. |
